# V2: Middleware + Blade Helpers for Experiment Variant Switching

## Motivation

Currently the consuming application must manually resolve experiment assignments in every controller that participates in an A/B test. This is repetitive, error-prone plumbing. Adding optional middleware + Blade helpers turns the common case into a one-liner while keeping the low-level `ResolveExperimentAssignment` action available for custom flows.

## Design principles

1. **Opt-in** — no existing code breaks. The new features are behind config flags and opt-in middleware registration.
2. **Middleware exposes, does not render** — middleware resolves the assignment and makes it available on the request. The view/controller still decides *what* to render.
3. **Backward compatible** — `ResolveExperimentAssignment`, `ProjectExperimentContextIntoSignalProperties`, etc. remain unchanged and fully usable.
4. **No hard dependency on Blade** — helpers are in a `BladeServiceProvider` that only registers if Blade is available.

## Feature breakdown

### 1. Config flags (`config/growth.php`)

```php
'features' => [
    'owner' => [/* existing */],
    'preset_modules' => [/* existing */],
    'experiment_middleware' => [
        'enabled' => false, // opt-in
    ],
    'blade_directives' => [
        'enabled' => false, // opt-in
    ],
],
```

### 2. Experiment middleware

**Class**: `AIArmada\Growth\Http\Middleware\ResolveExperiment`

**Registration**: auto-registered in the container as `growth.experiment`, available for `Route::middleware('growth.experiment:slug')` or manual registration.

**Logic**:

```php
final class ResolveExperiment
{
    public function handle(Request $request, Closure $next, string $experimentSlug): Response
    {
        $experiment = Experiment::query()
            ->where('slug', $experimentSlug)
            ->where('status', ExperimentStatus::Active)
            ->firstOrFail();

        $identity = $this->resolveIdentity($request);
        $session = $this->resolveSession($request);
        $anonymousId = $this->resolveAnonymousId($request);

        $assignment = app(ResolveExperimentAssignment::class)->handle(
            $experiment,
            identity: $identity,
            session: $session,
            anonymousId: $anonymousId,
        );

        $request->attributes->set('growth.experiment', $experiment);
        $request->attributes->set('growth.variant', $assignment->variant);
        $request->attributes->set('growth.assignment', $assignment);

        return $next($request);
    }

    private function resolveIdentity(Request $request): ?SignalIdentity
    {
        // Resolve from authenticated user -> external_id lookup
    }

    private function resolveSession(Request $request): ?SignalSession
    {
        // Resolve from session cookie -> SignalSession lookup
    }

    private function resolveAnonymousId(Request $request): ?string
    {
        // Resolve from request header or cookie (configurable key name)
    }
}
```

**Route usage**:

```php
Route::get('/sales', SalesPageController::class)
    ->middleware('growth.experiment:sales-page-test');
```

**Controller access**:

```php
class SalesPageController
{
    public function __invoke(Request $request)
    {
        $variant = $request->attributes->get('growth.variant');
        // or use a facade/helper
    }
}
```

### 3. Global helper & Facade

**Helper**: `experiment()` returns the current experiment context from the request.

```php
function experiment(): ?ExperimentContext
{
    $request = request();

    if (! $request instanceof Request) {
        return null;
    }

    $experiment = $request->attributes->get('growth.experiment');
    $variant = $request->attributes->get('growth.variant');
    $assignment = $request->attributes->get('growth.assignment');

    if (! $experiment instanceof Experiment || ! $variant instanceof Variant) {
        return null;
    }

    return new ExperimentContext($experiment, $variant, $assignment);
}

// Simple access:
experiment()->slug;        // 'sales-page-test'
experiment()->variantCode; // 'hero-a'
experiment()->isVariant('hero-a'); // bool
experiment()->assignmentId; // the sticky assignment UUID
```

**Facade** (optional, for testing):

```php
Growth::variantCode(); // 'hero-a'
```

### 4. Blade directives

Registered only when `growth.features.blade_directives.enabled` is true.

```php
// GrowthBladeServiceProvider::boot()

Blade::if('variant', function (string $code): bool {
    $context = experiment();

    return $context !== null && $context->isVariant($code);
});
```

Usage in views:

```blade
@variant('hero-a')
  <x-sales.hero-a />
@elsevariant('hero-b')
  <x-sales.hero-b />
@else
  <x-sales.default-hero />
@endvariant
```

Or for simple checks inside expressions:

```blade
@if(experiment()?->isVariant('hero-a'))
  <p>You're seeing the test variant</p>
@endif
```

### 5. Service provider registration

In `GrowthServiceProvider::packageRegistered()`:

```php
if (config('growth.features.experiment_middleware.enabled', false)) {
    $this->app->singleton(ExperimentContextResolver::class);
    $this->app->bind('growth.experiment_middleware', ResolveExperiment::class);
    $this->app->make(Router::class)->aliasMiddleware(
        'growth.experiment',
        ResolveExperiment::class,
    );
}
```

The Blade directives are registered in a separate `GrowthBladeServiceProvider` that is loaded from `GrowthServiceProvider::boot()` when enabled.

## ExperimentContext value object

```php
final readonly class ExperimentContext
{
    public function __construct(
        public Experiment $experiment,
        public Variant $variant,
        public Assignment $assignment,
    ) {}

    public function isVariant(string $code): bool
    {
        return $this->variant->code === $code;
    }

    public function isControl(): bool
    {
        return $this->variant->is_control;
    }

    public function variantCode(): string
    {
        return $this->variant->code;
    }

    public function assignmentId(): string
    {
        return $this->assignment->getKey();
    }

    public function experimentSlug(): string
    {
        return $this->experiment->slug;
    }

    public function toArray(): array
    {
        return [
            'experiment_id' => $this->experiment->getKey(),
            'experiment_slug' => $this->experiment->slug,
            'variant_id' => $this->variant->getKey(),
            'variant_code' => $this->variant->code,
            'assignment_id' => $this->assignment->getKey(),
        ];
    }
}
```

## Identity resolution strategy

The middleware needs a strategy to map the incoming request to a signals identity/session/anonymous ID. This should be configurable:

```php
// config/growth.php
'features' => [
    'experiment_middleware' => [
        'enabled' => false,
    ],
],

'http' => [
    'experiment_middleware' => [
        'subject_resolver' => null,
        'anonymous_id_source' => 'cookie',
        'anonymous_id_key' => 'sig_vid',
        'session_identifier_source' => 'cookie',
        'session_identifier_key' => 'sig_sid',
    ],
],
```

Default resolvers are provided, but the shipped configuration now hangs off `http.experiment_middleware.subject_resolver` plus the cookie/header settings shown above.

## Non-HTTP surfaces (queues, commands)

The middleware only covers HTTP. For jobs/commands, the app still uses `ResolveExperimentAssignment` directly with explicit owner context. This is by design — non-HTTP surfaces should be explicit about identity and owner scope.

## What this does NOT include

- **No rendering abstraction** — the app decides what to render per variant. No "page builder" or "variant templates" in growth.
- **No route registration** — the app registers its own routes. The middleware only acts on routes the app already owns.
- **No automatic experiment slug discovery** — the slug is explicit in the middleware argument. No magic route matching.
- **No Inertia/Livewire-specific helpers** — these belong in companion packages or the app layer.

## Implementation order

1. `ExperimentContext` value object
2. `experiment()` helper function
3. `ResolveExperiment` middleware + container alias
4. Config flags (`experiment_middleware.enabled`, `blade_directives.enabled`)
5. Identity/session/anonymous resolvers in middleware
6. Blade `@variant` / `@elsevariant` / `@endvariant` directives
7. Config docs and usage docs
8. Tests: middleware sets context, Blade renders correct block, helper returns null outside HTTP, identity resolution variations

## Migration path

Users of the current API continue as-is. New projects enable the features in config and start using middleware + Blade. The low-level `ResolveExperimentAssignment` is never removed.
